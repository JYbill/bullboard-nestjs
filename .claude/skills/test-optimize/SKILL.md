## 目标

测试要证明真实行为，选择能提供有效信心的最小测试类型，避免只验证测试准备代码。

## 检查流程

## 测试类型选择

1. **单元测试**：验证纯逻辑、校验规则、分支、错误处理、本地 service 行为。除非被测对象本身就是文件工具，否则不访问真实数据库、网络服务、队列、缓存、文件系统或第三方 API。
2. **集成测试**：验证代码和真实依赖、真实框架边界能否协同工作。优先使用真实依赖，不 mock 正在被集成验证的部分。共享基础设施默认只读；需要写入时必须有隔离数据和清理逻辑。
3. **端到端测试**：通过真实公开入口验证完整流程，例如 HTTP、CLI、消息入口或公开 SDK。使用隔离测试环境和最少必要数据，测试后清理或重建状态。

能用单元测试直接证明行为时，不升级成集成测试或端到端测试。

## 单元测试标准

单元测试应该优先验证：

- 输入到输出的有意义转换。
- 可能被错误实现的分支或规则。
- 边界条件，例如空输入、重复输入、缺失字段、非法类型、不支持状态、超时选择。
- 明确的错误消息或失败模式。
- 和协作者的交互，但前提是这个交互本身就是承诺给调用方的行为。

不要写只证明 JavaScript、TypeScript、JSON、测试框架或测试准备代码能工作的单元测试。

### 注释要求

所有单元测试的每个 `it`、`test`、`it.each()`、`test.each()` 块上方都要有中文注释，包含：

- `目的`：为什么要测这个场景。
- `总结`：这个测试最终保护了什么行为。

每个 `expect()` 断言块上方都要有中文注释，包含：

- `期待`：断言要观察到的输出、错误、调用或状态。

多个 `expect()` 共同验证同一件事时，可以共用一条紧贴的“期待”注释；验证不同内容时分别写。

```ts
// 目的：验证只提供成对字段的一侧时会抛出明确错误。
// 总结：缺少配对字段会触发配置完整性校验。
it("rejects incomplete paired field config", () => {
  writeInput([{ publicKey: "value" }]);

  // 期待：加载配置时抛出提示 publicKey 和 privateKey 必须成对填写的错误。
  expect(() => loadInput()).toThrow("publicKey 和 privateKey 必须同时填写或同时留空");
});
```

不要写空泛注释，例如“目的：测试 loadInput”“总结：测试正常运行”。

## 无意义断言处理

遇到 expected 直接来自 setup 的整体比较，通常删除或改写：

```ts
const input = { name: "example", enabled: true };
writeInput(input);

expect(readInput()).toEqual(input);
```

问题不是 `toEqual()`，而是断言只证明“写进去又原样读出来”。当 expected 是独立定义的，并体现归一化、转换、过滤、排序、校验或默认值规则时，`toEqual()` 可以保留。

成功路径只在能保护明确、可能回归的行为时保留；否则优先写聚焦的失败、边界或规则测试。

## 断言质量检查

新增或保留断言前，检查：

- 目标规则被移除时，这个断言会失败吗？
- 它检查的是调用方真正依赖的行为吗？
- expected 是独立选择的，还是从 setup 对象复制来的？
- 断言足够具体吗，失败时能定位回归点吗？
- `expect()` 上方的注释是否说明了期待观察到的具体内容？
- 测试名称描述的是行为，而不是实现细节吗？

大多数答案是否定的，就改写或删除。

## Mock 原则

mock 用来隔离被测单元，不要替代正在被验证的行为。

适合 mock：

- 单元测试中慢、不稳定、收费或与当前规则无关的协作者。
- 某个依赖调用本身就是要验证的契约。
- 为了确定性固定时间、随机数或进程环境。

不适合 mock：

- 集成测试声称验证数据库行为，却 mock 数据库。
- 声称验证 HTTP API 兼容性，却 mock HTTP 客户端。
- wrapper 没增加行为，只 mock 内部函数并断言它被调用。

## 测试数据和命名

使用能表达规则的通用 fixture，例如：`validInput`、`invalidInput`、`missingRequiredPair`、`emptyItems`、`duplicateItems`、`unsupportedState`。

## 评审已有测试

1. 按实际触达内容判断测试类型，不只看文件名。
2. 删除或改写只比较 setup 数据和返回数据的测试。
3. 成功路径测试只在能证明归一化、转换、兼容性或必要默认值时保留。
4. 对校验和保护逻辑增加聚焦的失败测试。
5. 用少量清晰断言替代大对象快照或整体对象比较。
6. 给每个单元测试块补“目的”和“总结”注释。
7. 给每个 `expect()` 断言块补“期待”注释。
8. 修改后运行最小相关测试命令；只有影响共享行为时才跑完整套件。

## 回复方式

说明测试修改时只报告：

- 适合的测试类型和原因。
- 删除或改写了哪些无意义断言。
- 当前保护的是哪个行为。
- 实际执行的验证命令和结果。
